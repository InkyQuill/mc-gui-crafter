# MCP Alpha Ergonomics and Template Defaults - Design Spec

## Overview

The 2026-05-23 alpha review used the `mc-gui-crafter` MCP app to create
Delightful Kitchen GUI prototypes. The workflow proved that the current MCP
surface can create and export useful screens, but it also exposed friction that
will keep LLM users from reliably building vanilla-style and mod-style GUIs:

- repeated slot creation is too slow;
- default project/template behavior is inconsistent;
- semantic schemas are underspecified;
- generated widgets such as buttons can exist as data but not render visually;
- preview/export accepts semantic mismatches that should be flagged before users
  wire generated code into a mod.

This spec turns the review findings into a focused alpha-fix package.

## Goals

- Make MCP GUI generation fast enough for slot-heavy layouts.
- Make template and empty-project behavior predictable for custom sizes.
- Include vanilla player inventory and hotbar grids by default where users expect
  a container-style screen.
- Let grids be created and moved as groups instead of one slot at a time.
- Render and export basic button visuals.
- Make MCP schemas and documentation precise enough that agents do not need trial
  and error for semantic group kinds, slot roles, or asset names.
- Add export preview warnings for semantic slot-count mismatches.
- Fix generated Java screen naming so `Screen` is not duplicated.

## Non-Goals

- Do not generate full container/menu business logic in this pass.
- Do not implement complete button interaction runtime behavior. Buttons need
  visible generated textures and metadata first.
- Do not replace individual slot elements with a single opaque inventory block.
  Grids should create real slot elements plus ordinary project groups.
- Do not redesign the whole editor grouping model. Add the minimum behavior
  needed for generated grids to be selected/moved as groups.
- Do not remove `asset_get_data_url`; compact list/import responses should
  coexist with explicit full-data retrieval.

## Reference Findings

### Open MCP Projects

The app currently has the review sessions open:

- `Delightful Kitchen - Auto Cutter`
  - `project_id`: `2219f3bf-6cb4-4945-aa35-98d6d962aefb`
  - fixed machine slots and progress arrow;
  - no player inventory/hotbar slots generated by default;
  - no project groups.
- `Delightful Kitchen - Food Pouch Basic Wide`
  - `project_id`: `3a367ba1-7fbc-43b9-aad0-77556f8ea988`
  - 18 pouch slots, six player inventory slots, two buttons;
  - buttons exist as `type: "button"` overlay elements but have no visible
    generated texture/render path;
  - no project groups;
  - imported/generated panel asset is named `textures/gui_panel.png`, which is
    surprising when importing `textures/generated/gui_panel.png`.

### Alpha Issues

1. `project_new` ignores requested dimensions when `template: "empty"` is used.
2. `project_save_as` is expected by docs and code, but was not visible in the
   MCP client's tool discovery during review.
3. `project_semantic_groups_update` schema is too vague and appeared as
   `semantic_groups: string[]` in client tooling, while runtime expects objects.
4. `SemanticGroupKind` and `SlotRole` accepted variants are under-documented.
5. Generated screen class names duplicate `Screen`.
6. Repeated `element_add` calls are too slow for player inventories and storage
   grids.
7. `asset_import` normalizes names unexpectedly and does not let the caller set
   the destination asset path.
8. `element_list` omits default `layer: "background"` fields, which is valid
   serialization but confusing for MCP consumers.
9. `asset_import` and `asset_list` return large base64 payloads by default.
10. Export preview does not validate semantic `slot_count` against matching
   elements.

## Design

### MCP Tool Surface

Add `element_add_many`.

Input:

```json
{
  "elements": [
    { "id": "slot_0", "type": "slot", "x": 8, "y": 84, "size": 18 },
    { "id": "slot_1", "type": "slot", "x": 26, "y": 84, "size": 18 }
  ],
  "project_id": "optional"
}
```

Behavior:

- validate every element first;
- reject duplicate ids against the project and within the request;
- record one history entry;
- return compact created elements and count.

Add `slot_grid_add`.

Input:

```json
{
  "id_prefix": "player_inv",
  "x": 8,
  "y": 84,
  "columns": 9,
  "rows": 3,
  "slot_size": 18,
  "spacing": 18,
  "slot_role": "player_inventory",
  "inventory_group": "player_inventory",
  "slot_index_start": 9,
  "group_id": "player_inventory_grid",
  "project_id": "optional"
}
```

Behavior:

- create `columns * rows` slot elements with 18px cadence by default;
- set `slot_index = slot_index_start + local_index`;
- set `slot_role`, `inventory_group`, and optional `scroll_binding`;
- create an ordinary project group when `group_id` is supplied;
- optionally create or update a matching semantic group when
  `semantic_group_kind` and `slot_count` are supplied;
- return `created_elements`, `group`, and any semantic group that was changed.

Accepted slot roles must be visible in the tool schema description:

- `machine`
- `player_inventory`
- `hotbar`
- `scrollable_inventory`
- `virtual_storage`
- `upgrade`
- `upgrade_settings`
- `filter`
- `ghost`
- `offhand`

Accepted semantic group kinds must be visible in tool schema descriptions:

- `fixed_slots`
- `virtual_slot_grid`
- `player_inventory`
- `hotbar`
- `upgrade_slots`
- `upgrade_panel`
- `search_field`
- `control_buttons`

Correct `project_semantic_groups_update` schema so `semantic_groups` is an
array of objects, not string-like values. The schema should include the enum
values above in field descriptions even if the local helper remains lightweight.

Ensure `tools/list` exposes `project_save_as`, `project_export_preview`,
`project_export`, `project_export_settings_update`, `project_semantic_groups_update`,
`element_add_many`, and `slot_grid_add`. Add a regression test that checks the
tool names in `tools/list`.

### Compact Assets

Change MCP `asset_import` response to compact metadata:

```json
{
  "name": "textures/generated/gui_panel.png",
  "width": 176,
  "height": 166,
  "bytes": 1234,
  "sha256": "..."
}
```

Add optional `name` or `asset_name` input to `asset_import`. When present, use
it as the destination asset path after validating that it ends in `.png` and is
relative. When omitted, keep the existing normalized `textures/<stem>.png`
behavior for compatibility.

Change MCP `asset_list` to return compact metadata without `data_url`.
`asset_get_data_url` remains the explicit tool for full binary payload retrieval.

Frontend API can keep full data URLs where the UI needs them, but MCP responses
should be compact by default. If backend commands are shared by UI and MCP, add
a compact flag or separate internal helper rather than breaking editor previews.

### Element List Defaults

Keep project serialization compact, but make MCP `element_list` less surprising:

- include effective `layer: "background"` for elements where the stored layer is
  default/omitted;
- do not change `.mcgui` save format just to satisfy MCP display.

This can be implemented with an MCP-only presentation helper that serializes
effective element values.

### Templates and Default Inventories

Fix `project_new` with `template: "empty"`:

- requested `width` and `height` must be respected;
- generated base assets must still be inserted;
- generated `gui_panel.png` must match the requested size.

Add reusable template helpers:

- `player_inventory_grid(x, y)` creates 27 slots, role `player_inventory`,
  group `player_inventory`, indices `9..35`, grouped as
  `player_inventory_grid`, semantic kind `player_inventory`.
- `hotbar_grid(x, y)` creates 9 slots, role `hotbar`, group `hotbar`, indices
  `0..8`, grouped as `hotbar_grid`, semantic kind `hotbar`.

Default machine/container templates should include both grids unless the
template is explicitly storage-only:

- `furnace`
- `advanced_machine`
- `fluid_tank`
- `brewing_stand`
- `anvil`
- `scrollable_inventory_machine`
- `custom_grid`

Chest templates are storage-first. They should not duplicate a player inventory
by default unless renamed or explicitly documented as a full container template,
but their storage slots should still have semantic metadata and a project group.

The base inventory placement for 176x166 screens should match vanilla cadence:

- player inventory 9x3: `x = 8`, `y = 84`, spacing `18`;
- hotbar 9x1: `x = 8`, `y = 142`, spacing `18`.

Templates taller than 166 can keep the same x cadence and choose y positions
that leave vanilla-style vertical separation.

### Group Behavior

Generated grids must be grouped automatically so users and LLMs can move them
as one object.

MCP:

- `slot_grid_add` creates a `Group` when `group_id` is supplied.
- Add or verify `group_create` can group any generated slot list.

Editor:

- If selecting an element in a group and moving it with the select tool already
  moves the group, keep existing behavior.
- If not, add group-aware movement for selected grouped elements in the canvas
  store/renderer path.
- Layer panel should continue to show group chips.

No separate "grid block" element is required in this pass.

### Buttons

Add generated button texture support.

Generated assets:

- `textures/generated/button.png` for a neutral 64x20 style button;
- optionally `textures/generated/button_disabled.png` and
  `textures/generated/button_hover.png` later.

Rendering:

- Pixi renderer draws `button` and `toggle_button` elements even without an
  imported asset.
- Button background uses generated Minecraft-like bevel styling.
- Button label uses the existing text rendering path, centered inside the
  button bounds.

Export:

- Static button background should be baked into the element's layer texture.
- Button label may remain runtime text/overlay metadata for now.
- Layout JSON keeps `type: "button"`, `content`, `binding`, dimensions, layer,
  and any semantic group metadata.

### Screen Class Naming

Introduce a normalized screen base class name:

- If user passes `AutoCutterGenerated`, exported screen file is
  `AutoCutterGeneratedScreen.java`.
- If user passes `AutoCutterGeneratedScreen`, exported screen file remains
  `AutoCutterGeneratedScreen.java`.

The constructor/class declaration should use the same normalized class name.
Client entrypoint comments should not append `Screen` twice.

Resource names may continue to derive from the raw/sanitized class name unless
tests show surprising output. The important fix is Java type/file naming.

### Semantic Preview Validation

Export preview should warn when semantic metadata is inconsistent.

Warnings:

- Semantic group with `slot_count` has fewer matching elements than declared.
- `player_inventory` group has fewer than 27 matching player inventory slots.
- `hotbar` group has fewer than 9 matching hotbar slots.
- `fixed_slots` group with `slot_count` has fewer matching slot elements.
- `virtual_slot_grid` group has fewer visible `virtual_slot_cell` or slot-like
  elements than `columns * visible_rows`.
- Scrollbar `target_group` does not match any semantic group id.
- Element `scroll_binding` references no scrollbar element.

Matching rules:

- First match by `inventory_group == semantic_group.id`.
- Also accept role-based matches for `player_inventory` and `hotbar` when the
  group ids are conventional.
- Count only visible elements.

These warnings should appear in `project_export_preview` and in the export
dialog preview. They should not block export.

## Testing

Rust tests:

- `project_new` with `template: "empty"`, custom width/height, and generated
  panel asset preserves requested dimensions.
- `tools/list` exposes all alpha MCP tools.
- `project_semantic_groups_update` schema describes object arrays and accepted
  semantic kinds.
- `element_add_many` validates atomically and records one history entry.
- `slot_grid_add` creates correct slot count, 18px cadence, slot indices, roles,
  project group, and optional semantic group.
- `asset_import` accepts explicit destination name and returns compact metadata.
- `asset_list` returns compact metadata without `data_url`.
- `element_list` includes effective default layer for MCP responses.
- Screen class naming does not duplicate `Screen`.
- Export preview warns for semantic group slot-count mismatch.
- Export preview warns for invalid scrollbar/scroll binding references.
- Generated button texture has expected dimensions and non-transparent pixels.
- Background/overlay export bakes button pixels.

Frontend checks:

- `pnpm check`.
- Button elements render in the editor.
- Button labels fit inside button bounds for common 64x20 and 80x20 buttons.
- Grouped generated grids can be selected and moved together.

MCP E2E:

- Create a custom-size empty project through MCP and verify size plus generated
  assets.
- Create a 9x3 player inventory with `slot_grid_add`.
- Create a 9x1 hotbar with `slot_grid_add`.
- Verify `group_list` includes both groups.
- Add button elements and verify they render/export visibly.
- Export with a class name ending in `Screen` and inspect generated Java paths.
- Create a semantic mismatch intentionally and verify preview warnings.

## Decisions

- Use real slot elements for generated grids, plus project groups. This keeps
  canvas selection/export behavior compatible with existing code.
- Keep compact `.mcgui` serialization, but make MCP read responses more explicit.
- Keep `asset_get_data_url` as the only normal way to retrieve base64 payloads.
- Prefer warning over blocking export for semantic mismatches during alpha.
- Add generated button visuals now; defer full runtime button click handling.
